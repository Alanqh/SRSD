# 一、命名空间能力拓展-Token管理联动DomeOS
## 1.1 需求背景与价值
目前，DomeOS平台已具备将部署（Deployment）或负载均衡（Loadbalancer）注册到外部Consul集群的能力（功能详情见https://gitbook.domeos.sohucs.com/consul_fu_wu_fa_xian.html）；同时，我们拥有独立的服务注册和发现系统，服务注册是其核心功能
然而，这两个系统目前是割裂的，当用户想要将DomeOS的部署或负载均衡注册到服务注册和发现平台上，需要手动、跨系统地配置Consul ACL Token权限，操作繁琐且容易出错，增加了运维成本和误操作风险。
本需求的核心价值：打通DomeOS与“服务注册和发现”系统，实现DomeOS资源一键注册到 “服务注册和发现”平台。用户在DomeOS侧完成服务注册后，即可完成“服务注册和发现”平台的注册和发现，提升操作效率与系统安全性，形成完整、流畅的服务治理闭环。
  
## 1.2 解决方案概述
  后端方案：
在“服务注册和发现”平台中，引入 “token” 概念。此token是Consul ACL Token在业务层面的抽象，用于桥接DomeOS服务注册与“服务注册和发现”平台的权限
  token位于命名空间下，一个命名空间下可以有多个token
  token可以关联相同命名空间下的服务
  一个服务可以关联多个token，同时一个token也可以关联多个服务
  当token关联服务时，该token便成为了在DomeOS侧完成服务注册的的“权限钥匙”，使用该token可以将DomeOS侧的部署或负载均衡注册到已关联的任意服务上
  当用户需要将DomeOS的部署或负载均衡注册到“服务注册和发现”平台上时，需要将“服务注册和发现”平台上的服务关联与对应token进行关联，DomeOS侧 ‘启用consul服务’ ‘名称’ 为”服务注册和发现“平台中 对应服务的”consul服务名称“。

产品设计：在命名空间下增加一个二级菜单：Token管理

## 1.3 验收标准
在“服务注册和发现”平台Web界面，显示服务对应的”consul服务名称“
在“服务注册和发现”平台Web界面，可以完成token的创建、查看、删除及与服务关联/解关联的全部操作。

# 三、新增虚拟服务
虚拟服务是指可以建一个服务，但他没有实例列表，而是关联其他同空间下的服务
然后可以配置一些流量分配的规则，访问虚拟服务时会访问到其他服务

产品设计：在服务下增加一个二级菜单：虚拟服务
